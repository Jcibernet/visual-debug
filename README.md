# visual-debug

> The critical UI/UX eye for coding agents. Inspect, measure, compare — without piling junk into the repo.

This README is written for you, the agent (Claude Code, Droid, Cursor). It's the
documentation you read when you find `visual-debug` inside a project. Direct
instructions, no fluff.

`visual-debug` is a single-file CLI that lets you **see and operate** a running
web app without a Playwright MCP server (and the ~3,500 context tokens it costs).
It runs over the shell, all state lives on disk, and you read JSON + SVG with
`cat`/`jq`.

The headline output **is not a screenshot**. It's a **layout SVG** (vector) plus
a **uxReport** (geometry and accessibility heuristics). You read structured text,
not pixels. PNG/JPEG are opt-in.

---

## What's new in v0.3.0

If you already know v0.2.0, here's what changed:

- **Ephemeral by default.** A run lives in a tmp dir and is deleted on exit.
  It no longer writes `.visual-debug/` into the repo unless you ask for it. The
  temp run path is printed to **stderr**.
- **PNGs are no longer generated by default.** To judge UI, read the `.layout.svg`
  and the `uxReport` — they are text and cost a fraction of the tokens. Request
  raster only with `--screenshots` or `--screenshot-on-issue`.
- **Persisting is opt-in and semantic**: `--persist-as <name>`, not timestamp
  accumulation.
- **Pipeable**: `--emit-manifest` sends the manifest to stdout; `--diff-against`
  accepts `-` (stdin).
- The old JSON did not change. Only new fields were added (`layout`, `uxReport`,
  `layoutSvg`) plus new diff categories (`layout`, `ux`). Everything is
  back-compatible.

## What it does

Three modes + one maintenance subcommand:

1. **URL** — `visual-debug <url>`: one-shot snapshot. Page map + layout SVG +
   uxReport (+ devtools dump: console, network, a11y, perf).
2. **Flow** — `visual-debug --flow -`: multi-step recipes you build inline and
   pipe over stdin. Click, fill forms, navigate, and snapshot at any step.
3. **Diff** — `visual-debug --diff a b`: compares two manifests and returns a
   verdict (`regression`/`changed`/`neutral`) with exit codes for your loop.
4. **Runs** — `visual-debug runs ...`: destructive maintenance of persisted runs
   (list, prune, clean). Lives apart so it never fires on its own.

---

## When to use it (decision tree)

```
Just want to inspect a view?
  → URL mode:  visual-debug <url>

Need to navigate/interact before seeing the state (login, form, tab)?
  → Flow mode: build the JSON inline and pipe it over stdin with --flow -

Iterating on a change and want to know if you broke something?
  → snapshot baseline (--emit-manifest > base.json)
    → edit the code
    → new snapshot | --diff-against base.json -
    → read the verdict

About to do a big refactor and want a named baseline?
  → --persist-as <name>  (the only case where persisting is worth it)
```

## Installation

Requires **Node 18+**.

```bash
npm i -g @jcibernet/visual-debug
```

The first run downloads Chromium via Playwright (~170MB). If you already have
one, point `--executable <path>` or the `VISUAL_DEBUG_CHROMIUM` env var at your
binary.

**First command, copy-paste, works with no prior setup:**

```bash
visual-debug https://example.com --emit-manifest | jq '.summary, .actions[0:3]'
```

That runs ephemeral (writes nothing to the repo), gives you the manifest over
stdout, and with `jq` you see the summary and the first interactables. The tmp
dir cleans itself up.

## URL mode — point inspection

```bash
# Ephemeral (default). The tmp dir path goes to stderr.
visual-debug http://localhost:3000/app

# Manifest to stdout for parsing with jq (summary goes to stderr):
visual-debug http://localhost:3000/app --emit-manifest | jq '.uxReport | keys'
```

Useful flags:

```bash
--viewport 375x812      # or --device "iPhone 14" for mobile
--dark                  # dark colorScheme
--wait "[selector]"     # wait for something to appear before capturing
--auth-storage <path>   # Playwright storageState (logged-in session)
--full-page             # only affects the raster (if you enable it)
```

## Flow mode — multi-step recipes over stdin

For when you need to click, fill forms, or reach a state before snapshotting.
Build the JSON inline and pipe it:

```bash
echo '{
  "name": "verify",
  "baseUrl": "http://localhost:3000",
  "steps": [
    { "navigate": "/app" },
    { "snapshot": "initial" },
    { "click": { "ref": 7 } },
    { "wait": "[data-step=detail]" },
    { "snapshot": "detail" }
  ]
}' | visual-debug --flow - --emit-manifest
```

**Targeting** (in order of preference):

```jsonc
{ "click": { "ref": 7 } }                          // by page-map index
{ "click": { "role": "button", "name": "Pay" } }   // by role + accessible name
{ "click": { "text": "Continue" } }                // by visible text
{ "click": { "testId": "submit" } }                // by data-testid
{ "click": "[data-action=pay]" }                   // raw CSS selector
```

The `ref`s come from the `actions` array of the previous snapshot. They are
recomputed against the **current** DOM on every step, so they never go stale
after re-renders.

Actions: `navigate`, `wait`, `snapshot`, `click`, `fill`, `type`, `press`,
`select`, `hover`, `scroll`, `eval`, `pause`. Per step: `optional: true`; per
flow: `continueOnError: true`.

To enable raster on just one snapshot: `{ "snapshot": "x", "screenshot": true }`.

## Diff mode — verify a change didn't break anything

```bash
# Classic form (two files):
visual-debug --diff baseline.json after.json --fail-on console,network,layout,ux

# Pipeable form (the candidate comes from stdin):
visual-debug http://localhost:3000/app --emit-manifest \
  | visual-debug --diff-against baseline.json - --fail-on layout,ux
echo "exit=$?"
```

- `verdict: neutral` → no relevant change.
- `verdict: changed` → there was a delta (layout/dom/perf/screenshot) with no
  errors or new UX findings.
- `verdict: regression` → console errors, failed requests, or new UX findings
  appeared.
- **Exit code `1`** if any `--fail-on` category fires. Valid categories:
  `console`, `network`, `perf`, `dom`, `screenshot`, `layout`, `ux`, `any`.
  Default: `console,network`. Use `any` for strict mode.

## Outputs: what to read and what to ignore

> **CORE RULE: read the manifest, the layout SVG, and the uxReport. DO NOT load
> the PNG/JPEG into context** unless `--screenshot-on-issue` generated one for a
> specific `severity:'error'` finding. The SVG + uxReport give you the layout and
> the problems in text, at a fraction of the token cost of an image.

### Manifest

The index of everything. Start here. It embeds what you need to decide the next
step without opening other files:

```bash
visual-debug <url> --emit-manifest | jq '.summary, .actions, .uxReport.tinyTapTargets'
```

- `summary` — quick counts (interactables, console errors, network failed,
  uxFindings per heuristic).
- `actions` — the first 50 interactables with `ref`/`role`/`name`/`selector`.
- `layout` — self-contained geometry (for diffing).
- `uxReport` — the heuristics (see below).
- `outputs` — paths to the run's files (in tmp if ephemeral).

### Page map (`<name>.map.json`)

Full inventory of interactables, forms, landmarks and headings, each with a
stable `ref`, `role`, accessible `name`, robust `selector` and `bbox`. The
manifest already embeds the first 50 in `actions`; open the map only if you need
more than 50 or the full forms/landmarks.

### Layout SVG (`<name>.layout.svg`)

The headline feature. A **vector** representation of the layout, generated from
the page map + bounding boxes (no rasterization). Read it with Read: it's text,
you "see" it without spending image tokens.

- One `<rect>` per interactable, positioned by its bbox, with `data-ref`,
  `data-role`, `data-name`.
- Color by role family: inputs blue, buttons green, links purple.
- Landmarks as background regions; headings as labeled rects.
- Elements flagged by the heuristics have a **dashed red border** and a
  `data-issue` attribute with the codes. Search for `data-issue=` in the SVG to
  jump straight to the problems.

### uxReport (heuristics)

An object in the manifest. Each heuristic is an array of
`{ ref?, selector?, code, message, severity }`. Collectors run in try/catch;
failures go to `uxReport.errors[]` and never break the run.

Geometry:

| Code | What it detects |
|---|---|
| `overflow` | the document overflows horizontally/vertically |
| `offscreen` | interactables outside the viewport |
| `tinyTapTargets` | targets < 44×44 (WCAG 2.5.5) |
| `overlaps` | interactables overlapping >50% (z-index/stacking) |
| `truncatedText` | text clipped with `text-overflow: ellipsis` |

Accessibility:

| Code | What it detects |
|---|---|
| `unlabeledInputs` | fields with no `<label>`/`aria-label`/`aria-labelledby` |
| `unnamedButtons` | buttons/links with an empty accessible name |
| `headingOrderJumps` | level jumps (h2 → h4) |
| `missingLandmarks` | missing `<main>`, `<nav>` or `<header>` |
| `imagesWithoutAlt` | `<img>` with no `alt` attribute (`alt=""` is valid) |
| `lowContrastPairs` | WCAG contrast < 4.5:1 (normal text) or < 3:1 (large) |

### Screenshot (opt-in, usually NOT)

Off by default. Enable it only if you truly need it:

```bash
--screenshots            # raster on every snapshot
--screenshot-on-issue    # raster ONLY if there is a severity:'error' finding
--screenshot-format webp # default; falls back to jpeg q70 on Playwright 1.x
```

If there is no PNG/JPEG in `outputs.screenshot`, **don't look for it**: it wasn't
generated on purpose. Read the SVG.

## Ephemeral vs persistent

**Strong bias toward ephemeral.** A run is a conversation with the page, not a
permanent artifact.

```bash
# Ephemeral (default): lives in tmp, deleted on exit. Doesn't touch the repo.
visual-debug <url>

# Named persistent: ONLY for baselines of a long job.
visual-debug <url> --persist-as login-baseline

# Auto-named persistent, with retention (default --keep 1):
visual-debug <url> --persist --keep 3
```

When to persist (`--persist-as`):

- You're about to do a **big refactor** and want a baseline to compare against
  later.
- You want to **capture a reproducible broken state** for a bug report (combine
  it with `--screenshot-on-issue`).

When **NOT** to persist:

- "Just in case." No. Use ephemeral.
- On every file save. No.

`--persist-as <name>` overwrites the dir if it already exists (you named it, you
own it). Persisted runs go to `.visual-debug/<name>/` — add `.visual-debug/` to
your `.gitignore` (don't commit it).

---

## The runs subcommand

Maintenance of persisted runs. **Destructive**, which is why it lives under a
subcommand: it never fires by accident.

```bash
# List persisted runs, with a fresh/stale/unknown check.
# (re-snapshots the manifest URL and compares against the saved page map)
visual-debug runs --list

# Delete runs whose DOM no longer matches the baseline (leaves 'unknown' alone).
visual-debug runs --prune-stale --yes

# Delete runs older than a duration (7d, 12h, 30m).
visual-debug runs --prune-older-than 7d --yes

# Delete EVERYTHING persisted.
visual-debug runs --clean --yes
```

Without `--yes` it asks for interactive confirmation (and refuses if there's no
TTY).

## JSON contracts (fields with stability guarantees)

Every field marked **stable** is API: it is not renamed or removed between minors.

### Snapshot manifest (`type: "snapshot"`)

| Field | Stable | Description |
|---|---|---|
| `type` | ✅ | `"snapshot"` |
| `name` | ✅ | run basename |
| `url` / `finalUrl` | ✅ | requested URL / final URL after redirects |
| `title` | ✅ | page `<title>` |
| `viewport` | ✅ | `[width, height]` |
| `outputs` | ✅ | paths to run files (`dom`, `console`, `network`, `a11y`, `perf`, `pageMap`, `layoutSvg`, `screenshot`) |
| `summary` | ✅ | counts: `console`, `network`, `perf`, `pageMap`, `uxFindings` |
| `actions[]` | ✅ | first 50 interactables: `{ ref, role, name, selector }` |
| `layout` | ✅ (v0.3) | `{ viewport, elements:[{ref,role,name,bbox}], landmarks }` |
| `layoutSvg` | ✅ (v0.3) | path to the `.layout.svg` |
| `uxReport` | ✅ (v0.3) | heuristics (see table above) + `errors[]` |
| `manifestPath` | ✅ | path to the manifest on disk |

### uxReport finding

| Field | Stable | Description |
|---|---|---|
| `code` | ✅ | heuristic identifier |
| `message` | ✅ | human-readable description |
| `severity` | ✅ | `'info'` \| `'warn'` \| `'error'` |
| `ref` | ✅ | interactable ref (if applicable) |
| `selector` | ✅ | CSS selector (if applicable) |

### Diff (`type: "diff"`)

| Field | Stable | Description |
|---|---|---|
| `verdict` | ✅ | `'neutral'` \| `'changed'` \| `'regression'` |
| `flags` | ✅ | booleans per category: `console`, `network`, `perf`, `dom`, `screenshot`, `layout`, `ux`, `any` |
| `console` | ✅ | `{ newErrors[], fixed[] }` |
| `network` | ✅ | `{ newFailures[], totalDelta }` |
| `perf` | ✅ | `{ fcpDelta, loadDelta }` |
| `dom` | ✅ | `{ added, removed, mutated }` |
| `layout` | ✅ (v0.3) | `{ added, removed, moved, movedRefs[] }` |
| `ux` | ✅ (v0.3) | `{ newFindings[], resolved[], newCount, resolvedCount }` |
| `summaryLine` | ✅ | one-liner of the diff |

### Page map (`<name>.map.json`)

| Field | Stable | Description |
|---|---|---|
| `interactables[]` | ✅ | `{ ref, role, name, selector, value, checked, disabled, bbox }` |
| `forms[]` | ✅ | `{ selector, action, method, fields[] }` |
| `landmarks[]` | ✅ | `{ role, selector, name, bbox }` |
| `headings[]` | ✅ | `{ level, text, selector, bbox }` |

## Approximate token costs

| Output | Typical size | Read it? |
|---|---|---|
| **manifest** | ~5–20 KB | **YES** — start here |
| **layout SVG** | ~10–50 KB | **YES** — to judge layout |
| **uxReport** (inside the manifest) | ~1–5 KB | **YES** — for UX/a11y problems |
| page map | ~5–30 KB | only if you need >50 interactables |
| PNG/JPEG | ~100–500 KB | **NO** — unless `--screenshot-on-issue` with an error |

**Recommendation: prefer SVG + uxReport, skip the PNG.** A 300KB image in context
costs orders of magnitude more than the vector SVG that says the same thing in
text.

## Known limitations

- **Layout SVG ≠ faithful render.** It's the *geometry* of interactables and
  landmarks, not the pixels. It doesn't catch color/image/typography problems
  beyond what the heuristics cover. For that specific need, `--screenshots`.
- **WCAG contrast is approximate.** It computes relative luminance over computed
  color/background-color, resolving the first ancestor with an opaque background.
  It doesn't handle gradients, background images, or `mix-blend-mode`.
- **WebP falls back to JPEG.** Playwright 1.x doesn't emit WebP;
  `--screenshot-format webp` uses JPEG q70 with a stderr warning.
- **`overlaps` is capped** to the first 250 interactables due to O(n²) cost.
- **The fresh/stale check in `runs --list`** re-snapshots the manifest URL; if
  the app isn't running, the run stays `unknown` (and `--prune-stale` won't touch
  it).
- **It doesn't replace Playwright MCP** for deeply interactive, step-by-step
  flows within a single turn. It is the cheapest default for inspection, flow
  navigation, regression, and UX auditing.

## How it works

`visual-debug.js` is a single ESM file on top of Playwright's headless Chromium.
It forces `QT_QPA_PLATFORM=xcb` to survive Wayland desktops with broken Qt
plugins. The page map and the uxReport come from a single DOM walk per snapshot;
each collector is wrapped in try/catch (a broken asset never breaks the run). The
layout SVG is produced by a pure function `renderLayoutSvg(pageMap, uxReport,
viewport)` — data → string, no DOM, no Playwright. The ephemeral run is cleaned
up with an idempotent handler on
`exit`/`beforeExit`/`SIGINT`/`SIGTERM`/`uncaughtException`.
